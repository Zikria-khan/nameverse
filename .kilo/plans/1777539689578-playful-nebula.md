# Plan to Reduce Static Page Generation and Avoid Vercel Limits

## Problem Analysis
The Next.js app is generating too many static pages during build, causing long build times and hitting Vercel limits (e.g., Edge Requests, Function Invocations, Build CPU Minutes). The main culprits are dynamic routes with `dynamic = 'force-static'` that either pre-generate many paths or use ISR on-demand generation for numerous variants.

Key pages causing issues:
- Name detail pages (`/names/[religion]/[slug]`) - potentially thousands of individual names
- List pages with pagination (`/names/religion/[religion]/[page]`, `/names/[religion]/origin/[origin]/[page]`, etc.) - many combinations with ISR
- Search pages (`/search/[term]`) - infinite possible terms

## Solution Overview
Change rendering strategy from static generation to dynamic (server-side) rendering for pages with high cardinality. This will:
- Eliminate build-time static generation for many pages
- Reduce edge function invocations by serving dynamically
- Shorten build times significantly
- Avoid pre-generating pages for rarely accessed content

## Implementation Steps

### 1. Update Name Detail Pages
**File:** `src/app/names/[religion]/[slug]/page.jsx`
- Change `export const dynamic = 'force-static';` to `export const dynamic = 'force-dynamic';`
- Remove `export const revalidate = 86400;` (not needed for dynamic rendering)
- Keep `export const dynamicParams = true;` (though default)

**Impact:** Prevents pre-generation of thousands of name pages, serves them dynamically instead.

### 2. Update List Pages with Pagination
**Files:**
- `src/app/names/religion/[religion]/[page]/page.jsx`
- `src/app/names/[religion]/origin/[origin]/[page]/page.jsx`
- `src/app/names/[religion]/categories/[category]/[page]/page.jsx`
- `src/app/names/[religion]/letter/[letter]/[page]/page.jsx`

For each:
- Change `export const dynamic = 'force-static';` to `export const dynamic = 'force-dynamic';`
- Remove `export const revalidate = 86400;`
- Remove or comment out `export async function generateStaticParams()` (prevents any static pre-generation)

**Impact:** Stops ISR-based static generation for pagination pages, serves all pages dynamically.

### 3. Update Search Pages
**File:** `src/app/search/[term]/page.jsx`
- Change `export const dynamic = 'force-static';` to `export const dynamic = 'force-dynamic';`
- Remove revalidate setting if present

**Impact:** Prevents caching of search result pages, serves dynamically.

### 4. Environment Configuration
**File:** `src/config/env.js`
- Ensure `NEXT_PUBLIC_BUILD_STATIC` is not set to 'true' (defaults to false)
- Keep low limits for static generation if needed

### 5. Testing and Monitoring
- Deploy and monitor build times
- Check Vercel analytics for reduced static generation
- Verify SEO impact (consider adding dynamic sitemap generation if needed)
- Monitor page load performance (dynamic rendering may be slightly slower but acceptable for many pages)

## Trade-offs and Considerations
- **SEO Impact:** Static pages provide better SEO; dynamic pages may have slower initial loads
- **Performance:** Dynamic rendering uses edge functions per request vs cached static pages
- **Costs:** May increase Vercel function invocations but reduce build costs
- **Mitigation:** Implement proper caching headers, CDN optimization, and consider hybrid approach for popular pages
